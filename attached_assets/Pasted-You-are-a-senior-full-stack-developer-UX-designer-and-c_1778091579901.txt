You are a senior full-stack developer, UX designer, and conversion copywriter. Build a production-ready MVP website and booking system for a paid kids’ item collection service in Cranbourne, Victoria, Australia.

Project name:
Second Life Kids

Primary goal:
Create a website with booking flow for a paid collection service where local parents can pay online to have unwanted children’s items collected from their home. The service is not a rubbish removal service. It is a convenience + sustainability service that gives reusable children’s items the best chance at a second life.

Tone and copy style:
Use warm, trustworthy, Australian English.
Use storytelling and outcome-focused copy.
Do not make the service sound like a charity.
Do not lead with donations.
Do not overpromise that everything avoids landfill.
Use words like “collection”, “items”, “reuse”, “second life”, “declutter”, “landfill”, “resourceful”, “simple”, and “no selling, no drop-offs, no stress.”
Avoid “trash” and use “rubbish” only when explaining what is not accepted.

Business model:
Customers pay for Trevor/Second Life Kids to collect unwanted children’s items from homes in Cranbourne and nearby suburbs.
After collection, items are sorted:
- A-grade reusable items may be listed for resale on the 0 to 17 app.
- B/C-grade reusable items may be bundled for responsible reuse/export partners.
- Unsold items after a 5-month cap may be removed from the app and moved through reuse/export channels.
- Recycling is used when appropriate.
- Landfill is only a last resort where there is no reasonable reuse/recycling option.

Important positioning:
This is a paid convenience service, not a free donation pickup.
The customer outcome is:
- clear the clutter
- no selling
- no drop-offs
- no need to be home
- items get a better outcome than sitting in storage or going straight to landfill

Tech stack:
Use:
- React + Vite + TypeScript for the frontend
- Tailwind CSS for styling
- Node.js + Express for the backend
- SQLite for the MVP database
- Stripe Checkout for payment
- Zod or equivalent validation

Do not store card details. All payment details must be handled by Stripe Checkout.

Required routes/pages:
1. / or /book-collection
   Main landing page and booking entry point.

2. /success
   Payment success page. It should retrieve the Stripe session status and show the confirmed booking details.

3. /cancel
   Payment cancelled page. It should let the user return to the booking form.

4. /admin
   Simple password-protected admin dashboard using ADMIN_PASSWORD from env.
   Admin should be able to:
   - View bookings
   - Filter by status
   - View booking details

   - Mark bookings as cancelled

5. /api/create-checkout-session
   Backend endpoint that creates the Stripe Checkout session.

6. /api/stripe-webhook
   Stripe webhook endpoint that listens for checkout.session.completed and marks the booking as paid.

7. /api/bookings
   Create/read booking records.

8. /api/cancel-request
   Allow customers to request cancellation using booking ID and email.

Stripe requirements:
Use Stripe Checkout in payment mode.
Currency: AUD.
Use automatic_payment_methods enabled.
Create one-time payment line items using price_data.
Pass booking ID, selected tier, pickup date, customer name, email, phone, suburb, postcode, and address in Stripe metadata.
Success URL:
APP_URL/success?session_id={CHECKOUT_SESSION_ID}
Cancel URL:
APP_URL/cancel?booking_id=<bookingId>

Use this general Stripe flow:
1. User completes booking form.
2. Backend creates a pending booking in SQLite.
3. Backend creates Stripe Checkout Session.
4. User is redirected to Stripe-hosted Checkout.
5. Stripe webhook confirms successful payment.
6. Booking status changes from PENDING_PAYMENT to PAID.
7. Customer sees success page.
8. Admin can see booking in dashboard.
